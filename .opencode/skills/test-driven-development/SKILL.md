# test-driven-development

这是一个 **兼容性 skill**。

目的不是强制完整教条式 TDD，而是为当前项目提供一个稳定、轻量、可执行的默认流程，避免运行时因缺少同名 skill 而失败。

## 适用场景

- 修复 bug
- 新增功能
- 修改已有行为且可通过测试验证

## 默认流程

1. 先确认行为边界
   - 改什么
   - 不改什么
   - 预期输入 / 输出 / 错误情况

2. 先找已有测试
   - 优先复用现有测试文件和测试风格
   - 没有相关测试时，再新增最小测试

3. 先写或更新能暴露问题的测试
   - 避免写空洞测试
   - 测试必须覆盖真实行为，不是占位断言

4. 做最小实现
   - 只修当前问题
   - 不借机大重构

5. 运行聚焦测试，再运行相关更广测试

## 明确禁止

- 不为了让测试通过而删除失败测试
- 不写 `expect(true).toBe(true)` 之类的空洞测试
- 不在 bugfix 时顺手重写半个模块

## 项目内原则

- 优先最小可验证改动
- 优先聚焦测试，不必每次都先跑全量
- 如果是纯文档或纯配置变更，不必机械套用 TDD
