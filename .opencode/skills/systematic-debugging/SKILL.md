# systematic-debugging

这是一个 **兼容性 skill**。

目的不是增加流程负担，而是提供一个稳定的默认排查方法，并避免运行时因缺少同名 skill 而失败。

## 适用场景

- 测试失败
- 页面或接口行为异常
- 本地与预期不一致
- 构建、类型检查、运行期报错

## 默认流程

1. 先复现
   - 明确输入、页面、命令、环境
   - 记录实际结果与期望结果的差异

2. 缩小范围
   - 先找最近改动
   - 先找最可能相关的模块
   - 优先读代码和现有测试，不要盲改

3. 验证假设
   - 一次只验证一个原因
   - 改动后立即运行最相关的检查

4. 做最小修复
   - 修根因，不修表面症状
   - 避免在未确认原因时“多改几处碰碰运气”

5. 回归验证
   - 先跑聚焦验证
   - 再跑相关更广验证

## 明确禁止

- 不做 shotgun debugging
- 不用随机改动碰运气
- 不在没定位原因时顺手大改结构
- 不靠 suppress type errors 逃避问题

## 项目内原则

- 先读，后改
- 先聚焦，后扩散
- 先证明，后宣称修复完成
